## Fork-Based Sessions Branch Strategy

이 프레임워크는 `/fork` 기능을 활용하여 **외부 연동 기술의 여러 경로를 동시에 탐색**합니다.

### 전략 개요

```
초기 설정
  ↓
[PHASE 1: 외부 연동 심층 조사 - 10개 Branch 병렬 실행]
  ├─ /fork Branch 1.1: MCP Server Specialist (풍부한 MCP 생태계 활용)
  ├─ /fork Branch 1.2: MCP Server Specialist (최소 MCP, 커스텀 서버 중심)
  ├─ /fork Branch 2.1: Local Tool Integration Expert (적극적 로컬 도구 활용)
  ├─ /fork Branch 2.2: Local Tool Integration Expert (최소 로컬 도구 의존)
  ├─ /fork Branch 3.1: API & Service Connector (클라우드/외부 서비스 연동)
  ├─ /fork Branch 3.2: API & Service Connector (완전 로컬/오프라인 운영)
  ├─ /fork Branch 4.1: Data Flow Architect (실시간 스트리밍 데이터 교환)
  ├─ /fork Branch 4.2: Data Flow Architect (배치 파일 교환)
  ├─ /fork Branch 5.1: Reliability & Fallback Engineer (Fail-fast 전략)
  └─ /fork Branch 5.2: Reliability & Fallback Engineer (Graceful degradation 전략)
  ↓
[PHASE 2: 연동 토론]
  ├─ /fork Branch 2.A: 기능 범위 극대화 우선 토론
  ├─ /fork Branch 2.B: 연동 안정성 우선 토론
  ├─ /fork Branch 2.C: 설정 단순성 우선 토론
  └─ /fork Branch 2.D: 자립성/독립성 우선 토론
  ↓
[PHASE 3: 연동 시나리오]
  ├─ /fork Branch 3.A: Full Integration (최대 연동, 최대 기능, 최대 복잡도)
  ├─ /fork Branch 3.B: Selective Integration (핵심 연동만, 균형)
  └─ /fork Branch 3.C: Self-Contained (최소 외부 의존, 최대 독립성)
  ↓
[PHASE 4: 조사 결과 통합]
  → 외부 연동 심층조사 종합 보고서
  → 연동 구성 레퍼런스
  → 추가 조사가 필요한 발견 사항
```

### Branch 사용의 목적

**조사 Phase에서의 Branch**
- 같은 연동 목적도 접근에 따라 복잡도, 안정성, 의존성이 전혀 달라짐을 검증
- 예: "기존 MCP 서버 활용" vs "커스텀 MCP 서버 개발" → 같은 목적인데 비용과 유연성이 다름
- 각 접근의 장단점을 구체적 구성 예시로 비교
- 탐색 중 범위 밖 발견은 파킹 로트에 포착

**토론 Phase에서의 Branch**
- 어떤 연동 철학을 우선시하냐에 따라 구성이 달라짐을 검증
- "기능 최대화" vs "안정성" vs "단순성" vs "자립성"의 경로
- 각 경로의 비용-효과를 파악

**시나리오 Phase에서의 Branch**
- 연동 수준에 따른 3가지 구성 경로 동시 검증
- 각 시나리오의 기능 범위, 안정성, 관리 부담을 비교

## 실행 제약 (Constraints)

**조사 단계 제약**
- 각 Teammate는 **근거 없는 연동 추천 금지** — 실제 작동 확인 또는 공식 문서 근거 필수
- 각 연동 대상에 대해 **실제 연결 가능 여부, 안정성, 설정 복잡도**를 반드시 평가
- "이 연동은 안 된다"는 결론은 **구체적 기술 제약**(미지원 프로토콜, 인증 불가 등)을 명시
- 각 Teammate는 다른 관점의 영향을 받지 말 것 (독립성 유지)
- **파킹 로트 규칙**: 조사 중 발견했으나 현재 Branch 범위를 넘는 사항은 파킹 로트에 기록

**토론 단계 제약**
- 토론은 **연동 구성의 비교**에 집중
- "좋아 보인다"는 주장 금지 → 안정성 수치, 설정 단계 수, 실패 빈도 등 구체적 근거 필수
- 한 연동 방식이 연속 3번 이상 강조되면, 대안에 명시적 발언 기회 제공

**시나리오 단계 제약**
- Full Integration: 가능한 모든 외부 연동 포함
- Selective Integration: 핵심 연동만, 나머지는 Claude Code 내장 기능으로 대체
- Self-Contained: 외부 의존 최소화, 오프라인 작동 가능
- 각 시나리오의 **외부 의존 수, 설정 복잡도, 실패 지점 수, 복구 전략** 명시

## 성공 지표 (Success Points)

### Phase 1 (연동 조사) 성공 지표

✅ **각 Teammate별 연동 분석이 구체적인가?**
- [ ] 각 연동 대상의 실제 연결 방법이 단계별로 명시되는가?
- [ ] 안정성, 설정 복잡도, 유지보수 부담이 평가되는가?
- [ ] 로컬 환경(Claude Code Max)에서의 실제 작동 여부가 확인되는가?

✅ **10개 Branch가 의미 있는 연동 대안을 제시하는가?**
- [ ] 각 Branch의 결론이 상충하는가? (충돌 없으면 분석 부족)
- [ ] 기능 범위, 안정성, 복잡도, 자립성 등 다각도 평가가 있는가?

✅ **연동 실패 시나리오가 분석되었는가?**
- [ ] 각 연동 지점의 실패 모드가 파악되었는가?
- [ ] 실패 시 워크플로우 전체에 미치는 영향이 분석되었는가?
- [ ] 복구/폴백 전략이 구체적인가?

### Phase 2 (토론) 성공 지표

✅ **연동 방식 간 트레이드오프가 명확한가?**
- [ ] 최소 3개 이상의 연동 방식 충돌 포착됨
- [ ] 각 충돌에 대해 구체적 차이(설정 단계, 실패 빈도, 복구 시간 등)가 명시됨

✅ **합의 영역이 명확해졌는가?**
- [ ] 4개 관점이 모두 동의하는 연동 구성이 파악됨

### Phase 3 (시나리오) 성공 지표

✅ **3가지 연동 시나리오가 현실적으로 구분되는가?**
- [ ] 각 시나리오의 외부 의존 수, 설정 복잡도, 실패 지점 수가 다른가?
- [ ] 각 시나리오의 구체적 연동 구성이 명시되었는가?

### Phase 4 (통합) 성공 지표

✅ **종합 보고서가 최종 PRD 작성 시 유의미한 참조 자료인가?**
- [ ] 연동 구성 레퍼런스가 실제 설계에 바로 사용 가능한 수준인가?
- [ ] 발견된 연동 한계·가능성이 명확히 정리되었는가?
- [ ] 추가 조사가 필요한 발견 사항이 정리되었는가?

## 팁 & Best Practices

**PHASE 1 (조사 Branch) 운영 팁**
- 각 Branch는 **완전히 독립적**으로 (다른 Branch를 보지 말 것)
- **실제 연결 테스트 결과**를 근거로 할 것 — 문서상 가능과 실제 작동은 다름
- MCP 서버는 **공식 지원과 커뮤니티 제공을 명확히 구분**할 것
- **보안 고려**: API 키, 토큰 등 민감 정보의 관리 방법을 반드시 포함
- **파킹 로트를 적극 활용** — 범위 밖 발견을 기록

**PHASE 2 (토론 Branch) 운영 팁**
- 연동 구성의 비교 시 **외부 의존 수, 설정 단계 수, 실패 지점 수**를 정량적으로 비교
- "이 연동이 꼭 필요한가?"를 항상 자문 — 불필요한 연동은 복잡도만 증가

**PHASE 3 (시나리오 Branch) 운영 팁**
- Full, Selective, Self-Contained는 **외부 의존도의 철학적 선택**
- 각 시나리오의 **오프라인 작동 범위**를 명시할 것 — 네트워크 환경이 항상 보장되지 않음
- Self-Contained 시나리오에는 **향후 연동 추가 시 확장 경로**를 반드시 포함

**PHASE 4 (최종 통합) 운영 팁**
- **연동 구성 레퍼런스가 가장 중요한 산출물** — MCP 서버 목록, 설정 방법, 폴백 체인이 실제 설계에 바로 사용 가능해야 함
- 발견된 연동 한계를 솔직하게 정리 — "문서에는 되지만 실제로는 안 된다"를 구분
- 추가 조사가 필요한 사항을 명확히 정리
