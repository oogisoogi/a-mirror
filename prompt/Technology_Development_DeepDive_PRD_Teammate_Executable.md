## Fork-Based Sessions Branch Strategy (확장 기능)

이 프레임워크는 `/fork` 기능을 활용하여 **기술과 구현 노하우의 여러 경로를 동시에 탐색**합니다.
이를 통해 기술 선택의 깊이 있고 견고한 기초를 확보합니다.

### 전략 개요

```
초기 설정
  ↓
[PHASE 1: 기술 심층 조사 - 10개 Branch 병렬 실행]
  ├─ /fork Branch 1.1: Platform Capability Researcher (최대 활용 관점)
  ├─ /fork Branch 1.2: Platform Capability Researcher (한계 인식 관점)
  ├─ /fork Branch 2.1: Configuration Architect (단순 설정 설계)
  ├─ /fork Branch 2.2: Configuration Architect (정밀 설정 설계)
  ├─ /fork Branch 3.1: Orchestration Engineer (경량 오케스트레이션)
  ├─ /fork Branch 3.2: Orchestration Engineer (고도 오케스트레이션)
  ├─ /fork Branch 4.1: Integration Specialist (최소 연동)
  ├─ /fork Branch 4.2: Integration Specialist (적극 연동)
  ├─ /fork Branch 5.1: Theory Foundation Expert (최신 에이전틱 이론 기반)
  └─ /fork Branch 5.2: Theory Foundation Expert (검증된 자동화 원칙 기반)
  ↓
[PHASE 2: 기술 토론]
  ├─ /fork Branch 2.A: 플랫폼 역량 극대화 우선 토론
  ├─ /fork Branch 2.B: 안정성/예측 가능성 우선 토론
  ├─ /fork Branch 2.C: 구현 속도 우선 토론
  └─ /fork Branch 2.D: 유지보수/확장성 우선 토론
  ↓
[PHASE 3: 기술 구현 시나리오]
  ├─ /fork Branch 3.A: Experimental (최신 기법, 높은 잠재력, 높은 리스크)
  ├─ /fork Branch 3.B: Pragmatic (검증+실험의 균형)
  └─ /fork Branch 3.C: Established (검증된 패턴만, 낮은 리스크)
  ↓
[PHASE 4: 조사 결과 통합]
  → 기술 심층조사 종합 보고서
  → 기술 구성 레퍼런스
  → 추가 조사가 필요한 발견 사항
```

### Branch 사용의 목적

**조사 Phase에서의 Branch**
- 같은 기술 영역에서도 다양한 접근이 가능함을 인정
- 예: "Claude Code Hooks를 최대한 활용" vs "Hooks의 한계를 인식하고 우회"
- 각 접근의 장단점을 명확히 검증
- 탐색 중 범위 밖 발견은 파킹 로트에 포착

**토론 Phase에서의 Branch**
- 어떤 기술 철학을 우선시하냐에 따라 결론이 달라짐을 검증
- "플랫폼 극대화" vs "안정성" vs "구현 속도" vs "유지보수성"의 경로
- 각 경로의 기술적 비용-효과를 파악

**시나리오 Phase에서의 Branch**
- 실험적 vs 균형 vs 검증된 기법 3가지 구현 경로 동시 검증
- 각 시나리오의 기술적 장단점, 복잡도, 안정성을 구체적으로 비교

## 실행 제약 (Constraints)

이 리서치 프로세스를 진행할 때 다음 제약을 유지하세요:

**조사 단계 제약**
- 각 Teammate는 **근거 없는 기술 추천 금지** (유행만으로는 안 됨)
- 해당 기술·기법을 사용하는 **실제 작동 사례 또는 공식 문서 근거를 최소 3개** 제시
- "이 기술은 안 된다"는 결론은 **기술적 제약의 구체적 지점**(어떤 API가 없는지, 어떤 기능이 미지원인지)을 명시
- 각 Teammate는 다른 관점의 영향을 받지 말 것 (기술적 객관성 유지)
- **Theory Foundation Expert는 반드시 이론적 근거를 명시**할 것
  - 원저자/연구자, 논문/발표, 발표년도
  - 이론과 실무의 갭을 명확히 할 것
- **파킹 로트 규칙**: 조사 중 발견했으나 현재 Branch 범위를 넘는 사항은 파킹 로트에 기록. 해당 사항이 외부 연동/사용자 행동/구조적 리스크 중 어디에 해당하는지 명시

**토론 단계 제약**
- 토론은 **기술 선택의 근거**에만 집중
- "좋아 보인다"는 주장은 안 됨 → "왜 좋은가" 기술적 근거 필수
- 한 기술이 연속 3번 이상 강조되면, 대안 기술에 명시적 발언 기회 제공
- **모든 권장사항은 실제 실행 테스트, 공식 문서, 또는 검증된 사례**로 뒷받침

**시나리오 단계 제약**
- Experimental: **최신 기법 위주** (최근 1-2년 내 등장, 커뮤니티 활발)
- Pragmatic: **검증 + 실험의 균형** (기본은 검증된 것, 핵심에만 실험적 기법 적용)
- Established: **검증된 패턴만** (다수의 사용자가 실제로 사용 중)
- 각 시나리오의 **구현 복잡도, 학습곡선, 디버깅 난이도** 명시
- 각 시나리오의 **구체적 기술 구성**을 명시

## 성공 지표 (Success Points)

이 기술 리서치 프로세스가 **성공했는지** 판단하는 기준:

### Phase 1 (기술 조사) 성공 지표

✅ **각 Teammate별 기술 분석이 깊이 있는가?**
- [ ] 단순 나열이 아니라, 각 기술·기법의 **구체적 장단점**이 명시되는가?
- [ ] 근거가 명확한가? (실제 실행 결과, 공식 문서, 검증된 사례)
- [ ] 우리 환경(로컬 컴퓨터, Claude Code Max)에 맞을지에 대한 **현실성 평가**가 있는가?

✅ **10개 Branch가 기술적으로 의미 있는 대안을 제시하는가?**
- [ ] 각 Branch의 결론이 **상충**하는가? (충돌 없으면 분석 부족)
- [ ] 학습곡선, 안정성, 확장성, 토큰 효율 등 **다각도 평가**가 있는가?

✅ **이론적 기초가 탄탄한가? (Theory Foundation Expert 평가)**
- [ ] **에이전틱 워크플로우 설계 이론**: 원칙이 명확한가?
- [ ] **최신 발전**: 새로운 개념·기법이 무엇인가?
- [ ] **이론과 실무의 갭**: "이론대로 하면 X인데, Claude Code 현실은 Y다"의 설명

✅ **기술적 한계와 가능성이 명확히 파악되었는가?**
- [ ] 예상보다 쉬운 것과 어려운 것이 구분되었는가?
- [ ] 현재 불가능한 것이 구체적 이유와 함께 정리되었는가?

### Phase 2 (기술 토론) 성공 지표

✅ **기술 간 트레이드오프가 명확히 드러났는가?**
- [ ] 최소 3개 이상의 **기술적 충돌** 포착됨
- [ ] 각 충돌마다 **구체적 비교**가 이루어졌는가?

✅ **합의 영역이 명확해졌는가?**
- [ ] 4개 관점이 **모두 동의하는 기술·기법** 파악됨
- [ ] 그 기술들의 조합이 **실제 작동하는 하네스**를 이룰 수 있는가?

### Phase 3 (시나리오) 성공 지표

✅ **3가지 기술 시나리오가 현실적으로 구분되는가?**
- [ ] 각 시나리오의 구현 복잡도, 학습곡선, 디버깅 난이도가 다른가?
- [ ] 각 시나리오의 기술적 장단점이 구체적으로 비교되는가?

✅ **각 시나리오의 구체적 기술 구성이 있는가?**
- [ ] CLAUDE.md 구조, Hooks 설계, 컨텍스트 관리 전략 등이 모두 정의됨
- [ ] 각 구성이 **실제로 작동**할 것 같은가?

### Phase 4 (통합) 성공 지표

✅ **종합 보고서가 최종 PRD 작성 시 유의미한 참조 자료인가?**
- [ ] 기술 구성 레퍼런스가 실제 설계에 바로 사용 가능한 수준인가?
- [ ] 발견된 기술적 한계·가능성이 명확히 정리되었는가?
- [ ] 추가 조사가 필요한 발견 사항이 정리되었는가?

## PHASE 1: 기술 심층 조사 (Fork-Based 병렬 탐색)

### 조사 Structure: 각 팀별 2개 Branch 병렬 실행

각 Teammate는 자신의 기술 영역 내에서 **상충하는 2가지 접근**을 동시에 탐색합니다.

#### **Platform Capability Researcher - 2개 Branch**

**Branch 1.1: Maximum Utilization (플랫폼 최대 활용)**

```
당신은 **Claude Code의 내장 기능을 최대한 활용**하는 관점의 기술 분석가입니다.

관점: "플랫폼이 제공하는 기능을 깊이 이해하면 외부 의존 없이 강력한 하네스를 만들 수 있다."

분석 대상: [리서치 주제]

분석 내용:

1. Claude Code 플랫폼의 현재 기능 심층 분석
   - Hooks 시스템: [이벤트 유형, 조건 분기, 실행 가능 범위]
   - Teammate/Fork: [병렬 실행 한계, 컨텍스트 격리 수준, 결과 통합 방법]
   - CLAUDE.md: [파싱 규칙, 계층 구조 지원 여부, 크기 한계]
   - .claude/ 디렉터리: [설정 파일 종류, 버전 관리, 이식성]
   - 세션 관리: [세션 지속성, 컨텍스트 윈도우 관리, 상태 유지 방법]

2. 각 기능의 실제 작동 확인
   - 공식 문서에서 명시된 기능 vs 실제 작동 차이: [구체적]
   - 문서화되지 않은 기능/동작: [있는가?]
   - 알려진 버그·제한·불안정 영역: [구체적]

3. 플랫폼 기능만으로 구현 가능한 워크플로우 패턴
   - 단순 파이프라인: [가능/불가]
   - 조건 분기 파이프라인: [가능/불가]
   - 반복/재시도 파이프라인: [가능/불가]
   - 멀티에이전트 오케스트레이션: [가능/불가]
   - 각 패턴의 구현 복잡도: [낮음/중간/높음]

결론:
- 플랫폼만으로 커버 가능한 범위: [구체적]
- 최대 활용의 강점: [구체적 3가지]
- 최대 활용의 한계: [어디서 막히는가?]
- 이 관점에서 필수 기술 구성 요소: [Top 3]

🅿️ 파킹 로트: [범위 밖 발견 사항 — 해당 심층조사 카테고리 명시]
```

**Branch 1.2: Limitation-Aware (한계 인식 관점)**

```
당신은 **Claude Code의 한계를 명확히 파악하고 우회 전략을 설계**하는 관점의 기술 분석가입니다.

관점: "플랫폼의 한계를 정확히 알아야 현실적인 설계가 가능하다."

분석 대상: [리서치 주제]

분석 내용:

1. Claude Code 플랫폼의 구조적 한계 심층 분석
   - 컨텍스트 윈도우 한계: [크기, 소진 속도, 복구 방법]
   - 토큰 소비 한계: [구독 한도, 워크플로우당 소비량, 최적화 여지]
   - 병렬 실행 한계: [동시 Teammate 수, 리소스 경합]
   - Hooks 한계: [지원하지 않는 이벤트/조건, 실행 환경 제약]
   - 상태 관리 한계: [세션 간 상태 유지 불가, 파일 의존]

2. 각 한계의 실제 영향도
   - 어떤 워크플로우 패턴에서 문제가 되는가?: [구체적]
   - 어떤 규모에서 문제가 되는가?: [구체적]
   - 예측 가능한 한계 vs 예측 불가능한 한계: [분류]

3. 한계 우회 전략
   - 컨텍스트 소진 우회: [상태 파일 분할, 요약 체인 등]
   - 토큰 한도 우회: [프롬프트 최적화, 배치 처리 등]
   - Hooks 한계 우회: [외부 스크립트 연동, 워치독 패턴 등]
```

## 팁 & Best Practices

### Branch 전략을 효과적으로 운영하기

**PHASE 1 (조사 Branch) 운영 팁**
- 각 Branch는 **완전히 독립적**으로 (다른 Branch를 보지 말 것)
- 기술 분석의 **근거(실행 테스트, 공식 문서, 검증된 사례)**를 반드시 포함
- "모든 Branch가 추천하는 기술·기법"을 찾을 것 (진정한 필수 기술)
- **파킹 로트를 적극 활용** — 범위 밖 발견을 기록

**Theory Foundation Expert Branch 운영 팁**
- 이론 분석은 **단순한 설명이 아니라 원칙적 근거**를 제시할 것
- **에이전틱 워크플로우 특유의 이론**과 **검증된 자동화 원칙** 양쪽 모두 다룰 것
- **이론과 현실의 갭**을 솔직하게 인정할 것
  - 예: "ReAct 패턴은 이론적으로 우아하지만, Claude Code의 컨텍스트 한계에서 어떻게 적용하는가?"
- **참고 자료 명시**: 논문/저자/발표년도를 구체적으로

**PHASE 2 (토론 Branch) 운영 팁**
- 각 Branch는 **특정 가치를 극단까지 밀어본다**
- 4개 Branch 결과를 비교할 때 **기술별 우선순위 변화**를 표로 정리할 것


**PHASE 3 (시나리오 Branch) 운영 팁**
- Experimental, Pragmatic, Established는 **기술 철학의 선택**
- 구현 복잡도, 학습곡선, 디버깅 난이도를 구체적으로 비교할 것

**PHASE 4 (최종 통합) 운영 팁**
- **기술 구성 레퍼런스가 가장 중요한 산출물** — 실제 설계에 바로 참조 가능해야 함
- 발견된 기술적 한계와 가능성을 솔직하게 정리
- 추가 조사가 필요한 사항을 명확히 정리

### 기술 리서치 특화 팁

- **실제 실행 테스트 우선**: "될 것 같다"는 추상적 주장 금지, 실행 결과로 증명
- **Claude Code 공식 문서 기준**: 비공식 정보보다 공식 문서와 실제 동작을 신뢰
- **우리 환경과의 적합성**: 대규모 팀의 사례가 아니라 "Claude Code Max 구독자 1인이 로컬에서 운용하는 상황"에 맞는가?
- **토큰 효율 항상 고려**: 어떤 기술 구성이든 토큰 소비량을 예측하고 구독 한도와 비교
