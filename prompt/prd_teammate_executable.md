# PRD 1차 리서치 - Teammate Execution Framework

> **이 문서의 목적**: 가장 넓은 그물을 던지는 1차 광범위 리서치 단계. 이 문서의 산출물은 최종 PRD가 아니라 "1차 PRD 초안 + 후속 심층조사 실행 계획"이다. 이후 기술 심층조사, 외부 연동 심층조사 등 별도의 전문 심층리서치를 거쳐, 모든 결과를 통합한 최종 PRD를 완성한다.

## 초기 설정 (당신이 입력할 것)

**제품명**: [여기에 입력]
**주요 사용 시나리오**: [이 하네스가 자동화할 핵심 작업 유형을 입력]
**해결하려는 워크플로우 문제**: [현재 수동으로 하고 있거나 비효율적인 작업 흐름을 입력]

## Fork-Based Sessions Branch Strategy (확장 기능)

이 프레임워크는 `/fork` 기능을 활용하여 **각 Phase에서 여러 경로를 동시에 탐색**합니다.
이를 통해 더 깊이 있는 리서치, 철저한 토론, 견고한 수렴을 이룹니다.

### 전략 개요

```
초기 설정
  ↓
[PHASE 1: 조사]
  ├─ /fork Branch 1.1: Workflow Architect (가정 A)
  ├─ /fork Branch 1.2: Workflow Architect (가정 B)
  ├─ /fork Branch 2.1: Scenario Explorer (가정 A)
  ├─ /fork Branch 2.2: Scenario Explorer (가정 B)
  ├─ /fork Branch 3.1: Operator Analyst (가정 A)
  ├─ /fork Branch 3.2: Operator Analyst (가정 B)
  ├─ /fork Branch 4.1: Sustainability Strategist (가정 A)
  └─ /fork Branch 4.2: Sustainability Strategist (가정 B)
  ↓
[PHASE 2: 토론]
  ├─ /fork Branch 2.A: 워크플로우 구조 우선 토론
  ├─ /fork Branch 2.B: 사용 시나리오 커버리지 우선 토론
  ├─ /fork Branch 2.C: 운용성/사용자 경험 우선 토론
  └─ /fork Branch 2.D: 지속 가능성/효율 우선 토론
  ↓
[PHASE 3: 수렴]
  ├─ /fork Branch 3.A: Deep Automation (고도 자동화)
  ├─ /fork Branch 3.B: Selective Automation (선택적 자동화)
  └─ /fork Branch 3.C: Minimal Automation (최소 자동화)
  ↓
[PHASE 4: 통합]
  → 3개 시나리오 비교
  → 1차 PRD 초안 확정
  → 후속 심층조사 실행 계획 산출
```

### Branch 사용의 목적

**조사 Phase에서의 Branch**
- 같은 관점에서도 다양한 가정이 가능함을 인정
- 각 가정이 타당한 근거를 가질 수 있음을 검증
- 탐색 중 범위 밖 발견은 파킹 로트에 포착

**가정 축 설정 지침**: 프로젝트 시작 시 다음 후보 중에서 가장 의미 있는 1개를 선택한다.
- "Claude Code 단독 완결 vs 외부 도구 연동 전제"
- "단일 사용자 전용 vs 팀 공유 가능 구조"
- "단일 프로젝트 집중 vs 멀티 프로젝트 동시 관리"
- "최소 자동화(사용자 개입 많음) vs 최대 자동화(완전 자율 실행)"

어떤 축을 선택하든 2개의 대립 가정을 강제로 탐색하게 되어 그물의 넓이가 보장된다.

**토론 Phase에서의 Branch**
- 어떤 관점을 가장 우선시하냐에 따라 결론이 달라짐을 검증
- "워크플로우 구조 우선" vs "시나리오 커버리지 우선" vs "운용성 우선" vs "지속 가능성 우선"의 각 경로가 다른 PRD를 만듦
- 각 경로의 장단점 파악

**수렴 Phase에서의 Branch**
- 자동화 범위와 구조적 복잡도를 기준으로 3가지 시나리오 동시 검증
- "만약 우리가 더 고도로 자동화했다면?" "더 최소한으로 시작했다면?"을 동시에 평가
- 각 시나리오가 4개 카테고리의 후속 조사 항목을 출력하여 심층조사 단계와 연결

## 실행 제약 (Constraints)

이 프로세스를 진행할 때 다음 제약을 유지하세요:

**조사 단계 제약**
- 각 Teammate는 **근거 없는 주장 금지** (추측/직관만으로는 안 됨)
- 대안 워크플로우 도구·방법론을 조사할 때는 해당 도구가 실제로 로컬 환경에서 작동하는지 **실행 가능성을 확인**해야 함
- Claude Code의 실제 실행 한계(컨텍스트 윈도우 크기, 토큰 소비 속도, 병렬 Teammate 수 제한, Hooks의 조건 분기 지원 범위)를 **반드시 명시**해야 함
- "불가능하다"는 결론을 낼 때는 **기술적 제약의 구체적 지점**(어떤 API가 없는지, 어떤 기능이 미지원인지)을 명시해야 함
- 각 Teammate는 다른 관점의 영향을 받지 말 것 (독립성 유지)
- **파킹 로트 규칙**: 조사 중 발견했으나 현재 Branch의 관점이나 범위를 넘는 사항은 파킹 로트에 기록한다. 발견 내용, 발견한 Branch, 해당하는 후속 조사 카테고리(기술/외부연동/사용자행동/구조적리스크)를 명시한다.

**토론 단계 제약**
- Moderator는 중립 입장 유지 (특정 팀의 편 들지 않기)
- 한 관점이 연속 3번 이상 강조하면, 반대 관점에 명시적 발언 기회 제공
- "동의한다"만으로는 안 됨 → "왜 동의하는가" 근거 포함 필수
- 충돌이 없으면 안 됨 (4개 관점이 모두 일치하면 토론 깊이가 부족한 신호)
- **파킹 로트 규칙**: 토론 중 발견된 범위 밖 사항도 동일하게 파킹 로트에 기록한다.

**수렴 단계 제약**
- Green Zone: **4개 관점 모두 동의** (3개만 동의하면 Yellow Zone)
- Yellow Zone: **명확한 트리거 조건** 정의 필수 ("언제 포함될 것인가?")
- Red Zone: **재검토 시점 명시** ("6개월 후" 또는 "~이면 검토")
- 구성 요소는 최대 15개 이내로 집약 (산만해지면 안 됨)
- **각 시나리오는 반드시 4개 카테고리의 후속 조사 항목을 출력**해야 함

## 성공 지표 (Success Points)

이 리서치 프로세스가 **성공했는지** 판단하는 기준:

### Phase 1 (조사) 성공 지표
✅ **각 Teammate별 결론이 구체적인가?**
- [ ] 워크플로우 구성 요소(Hook, Teammate, Pipeline, Trigger 등)가 구체적인가 (예: "자동화" ❌ → "커밋 시 자동 린트 + 테스트 실행 Hook" ✅)
- [ ] 근거가 명시되어 있는가 (예: "Claude Code Hooks 공식 문서 기준...", "실제 실행 테스트 결과...")
- [ ] 우선순위가 명확한가 (Top 3이 명확히 도출됨)

✅ **4개 관점이 서로 다른 각도에서 봤는가?**
- [ ] Workflow Architect만 강조하는 구성 요소가 있는가? (파이프라인 구조만 보는 것)
- [ ] Scenario Explorer만 강조하는 시나리오가 있는가? (커버리지만 보는 것)
- [ ] Operator Analyst만 조심하는 마찰 지점이 있는가? (사용성 문제)
- [ ] Sustainability Strategist가 우려하는 구조적 한계가 있는가? (지속 가능성 문제)
- → 모두 있으면 ✅ 다각적 분석 성공

### Phase 2 (토론) 성공 지표
✅ **충돌이 명확히 드러났는가?**
- [ ] 최소 3개 이상의 의견 불일치 포착됨
- [ ] 각 불일치마다 "왜 다른가"의 근거가 명시됨
- [ ] 단순 선호도 차이가 아니라 **논리적 충돌**임

✅ **합의 영역이 명확해졌는가?**
- [ ] 4개 관점이 **모두 동의**하는 구성 요소가 최소 5개 이상
- [ ] 그 구성 요소들이 핵심 워크플로우 문제를 **정말 해결**하는가?
- [ ] Green Zone 구성 요소들이 "이것들만으로도 작동하는 하네스가 되나?"의 답이 "YES"

✅ **위험 가정이 포착되었는가?**
- [ ] Top 5 위험 가정이 명시됨
- [ ] 각 가정마다 "검증 방법"이 구체적임 (예: "실제 실행 테스트", "Claude Code API 문서 확인")
- [ ] 위험 가정이 "워크플로우 전체를 무너뜨릴 수 있는" 수준의 것들임

### Phase 3 (수렴) 성공 지표
✅ **1차 PRD 초안이 실행 가능한가?**
- [ ] Green Zone 구성 요소들의 합계 구현 복잡도가 명시되고 **현실적**인가?
- [ ] 각 구성 요소의 우선순위 순서가 명확한가?
- [ ] 후순위로 미룬 구성 요소들이 **정말 미룰 수 있는** 것들인가?

✅ **모든 팀이 서명할 수 있는가?**
- [ ] Workflow Architect: "이 구조로 워크플로우가 안정적으로 실행될 수 있다" → 동의하는가?
- [ ] Scenario Explorer: "이 구성으로 핵심 사용 시나리오가 커버되는가?" → 동의하는가?
- [ ] Operator Analyst: "사용자가 이 설정 복잡도를 수용할 수 있는가?" → 동의하는가?
- [ ] Sustainability Strategist: "이 구조가 지속적으로 운영·확장 가능한가?" → 동의하는가?
- → 모두 "동의"해야 ✅

✅ **후속 심층조사 연결이 되어 있는가?**
- [ ] 각 시나리오에서 4개 카테고리(기술/외부연동/사용자행동/구조적리스크)의 후속 조사 항목이 출력됨
- [ ] 파킹 로트 항목이 후속 조사 카테고리와 병합됨
- [ ] 각 후속 조사 항목에 "이것이 검증되지 않으면 PRD의 어떤 결정이 흔들리는가"가 명시됨

## PHASE 1 Execution: Fork-Based 병렬 조사 구조

### 단계 1: 각 Teammate의 2개 Branch 독립 실행

이 단계에서는 **각 Teammate가 자신의 관점 내에서 상충하는 2가지 가정**을 동시에 탐색합니다.
가정 축은 초기 설정에서 선택한 것을 사용합니다. 아래는 "Claude Code 단독 완결 vs 외부 도구 연동 전제" 축을 예시로 사용합니다.

#### **Workflow Architect Branch A: Self-Contained (Claude Code 단독 완결)**

```
당신은 이 하네스의 워크플로우 구조를 **Claude Code 생태계 안에서 완결**되는 방향으로 설계하는 관점입니다.

핵심 가정: "외부 도구 없이 Claude Code의 내장 기능(Hooks, Teammates, /fork, CLAUDE.md, .claude/ 등)만으로 모든 워크플로우를 구현할 수 있다"

분석 내용:
1. 워크플로우 실행 구조
   - 파이프라인의 기본 단위: [태스크? 세션? 프롬프트 체인?]
   - 분기·반복·실패 복구 구조: [Claude Code 내장 기능으로 어떻게 구현하는가?]
   - 상태 저장 방식: [파일 기반? 메모리? .claude/ 디렉터리 활용?]

2. 설정 아키텍처
   - CLAUDE.md 구성 전략: [단일 파일? 계층적 분할?]
   - .claude/ 디렉터리 설계: [어떤 설정을 어디에 두는가?]
   - Hooks 활용 방식: [어떤 이벤트에 어떤 자동화를 거는가?]

3. 멀티 에이전트 실행
   - Teammate 간 컨텍스트 격리 전략: [어떻게 독립성을 보장하는가?]
   - 병렬 실행 시 상태 충돌 방지: [구체적 방법]
   - 결과 통합 메커니즘: [각 Teammate의 산출물을 어떻게 합치는가?]

결론:
- Claude Code 단독으로 구현 가능한 워크플로우 범위: [구체적]
- 단독 완결의 강점: [구체적 3가지]
- 단독 완결의 한계: [어디서 막히는가?]
- 이 구조에서 반드시 포함되어야 할 구성 요소: [Top 3]

🅿️ 파킹 로트: [이 Branch 범위 밖이지만 후속 조사가 필요한 발견 사항을 기록]
```

#### **Workflow Architect Branch B: Integrated (외부 도구 연동 전제)**

```
당신은 이 하네스의 워크플로우 구조를 **외부 도구와의 연동을 전제**로 설계하는 관점입니다.

핵심 가정: "Claude Code만으로는 한계가 있으며, MCP 서버, 외부 API, 로컬 스크립트 등과 연동해야 완전한 워크플로우가 된다"

분석 내용:
1. 워크플로우 실행 구조
   - 파이프라인의 기본 단위: [외부 도구 호출을 포함한 복합 단위]
   - 외부 연동 지점: [어떤 단계에서 외부 도구가 개입하는가?]
   - 오류 처리: [외부 도구 실패 시 폴백 전략]

2. 연동 아키텍처
```

## 당신의 임무: 구조화된 토론 진행

### STEP 1: 합의 영역 파악 (Overlap Analysis)


질문: "어디서 관점이 충돌하는가?"

아래 4가지 패턴을 확인:

**패턴 1: "구조적으로 필요한" vs "사용자가 감당 가능한"**
- Workflow가 강조하는데 Operator가 복잡도를 우려하는 구성 요소?

- 해결책: [절충안]

**패턴 2: "시나리오 커버를 위해 필요한" vs "지속 가능한"**
- Scenario가 강조하는데 Sustainability가 토큰/유지보수 부담을 우려하는 구성 요소?
- 이유: [Sustainability 관점 기술]
- 해결책: [최적화 방안?]

**패턴 3: "워크플로우 구조" vs "운용성"**
- Workflow의 이상적 구조 vs Operator의 실제 사용 가능 수준
- 우리가 단순화하거나 추상화할 수 있는가?
- 대응: [구체적 액션]

**패턴 4: "기능 범위" vs "효율성"**
- 더 많은 시나리오 커버 vs 토큰·유지보수 효율
- 우선순위: [무엇이 먼저인가?]

### STEP 4: 위험 가정 포착 (Riskiest Assumptions)

- Operator: "이 가정이 틀리면 사용자가 설정을 포기한다"
- Sustainability: "이 가정이 틀리면 한 달 만에 토큰이 바닥난다"

**형식:**

## 팁 & Best Practices

### Branch 전략을 효과적으로 운영하기

**PHASE 1 (조사 Branch) 운영 팁**
- 각 Branch는 **완전히 독립적**이어야 함 (다른 Branch의 결과를 보지 말 것)
- 선택한 가정 축의 **2개 대립 가정**을 명확히 할 것
- 모든 결론에 **근거(Claude Code 공식 문서, 실행 테스트, 실제 사례)**를 첨부할 것
- "모든 Branch가 동의하는 구성 요소"를 찾을 것 (그것이 진정한 필수 구성 요소)
- **파킹 로트를 적극적으로 활용**할 것 — 범위 밖 발견을 버리지 않고 포착

**PHASE 2 (토론 Branch) 운영 팁**
- 각 Branch는 **특정 관점을 극단까지 밀어본다** (워크플로우 우선이면 정말 구조만 봄)
- 이렇게 하면 "어떤 관점을 1순위로 할 때 어떤 결과가 나오는가"를 명확히 봄
- 4개 Branch의 결과를 비교할 때 "구성 요소별 우선순위 변화"를 표로 정리할 것
- "어떤 관점 조합이 이상적인가?"를 발견할 것

**PHASE 3 (시나리오 Branch) 운영 팁**
- Deep, Selective, Minimal은 **단순히 많고/적음이 아니라 자동화 철학의 선택**
- 각 시나리오는 "이 방향으로 가면 6개월 후 이렇게 되어 있을 것"을 명확히 보여줄 것
- **반드시 4개 카테고리의 후속 조사 항목을 출력**할 것 — 이것이 다음 단계의 입력
- 선택 기준을 명확히 할 것 (워크플로우 빈도, 사용자 숙련도, 토큰 예산, 확장 계획 등)

**PHASE 4 (최종 통합) 운영 팁**
- 3개 Scenario 중에서 "1등"을 고르는 것이 아니라, "우리 상황에 맞는" 것을 고르기
- "이 Scenario를 택하지 않은 이유는?"을 기록할 것
- **후속 심층조사 실행 계획이 가장 중요한 산출물** — 이것 없이는 최종 PRD로 갈 수 없음
- 1차 PRD 초안에 "심층조사 전 초안"임을 명시할 것 — 이것은 확정이 아니라 가설

### 일반 운영 팁 (유지)

조사 단계에서
- 각 Teammate가 "근거"를 명확히 하도록 강조
- 추상적 답변은 "구체적 예시를 3가지 주세요"로 재요청
- 충돌하는 의견이 나오면 좋음 (건강한 신호)

토론 단계에서
- Moderator가 중립적이어야 함
- 한 팀의 의견이 독주하면 "다른 관점은?"으로 균형 조절
- "동의합니다"는 약한 신호 → "왜 동의합니까?"로 깊이 있게

수렴 단계에서
- Green Zone은 **보수적으로** (모두 동의한 것만)
- Yellow Zone은 **명확한 기준**으로 (언제 포함될 것인가?)
- Red Zone도 **언제 재검토할 것인가** 명시
