# Coding & Implementation Deep-Dive Framework
## 연관 핵심 코딩 기술 집중 심층조사 — 로컬 에이전틱 워크플로우 하네스

---

> **이 문서의 목적**: 로컬 에이전틱 워크플로우 하네스를 **코드 수준에서 어떻게 구현하는가**에 대한 독립적이고 깊이 있는 조사를 수행한다. 1차 PRD 리서치, 기술 심층조사 등 다른 심층조사와 함께, 최종 PRD를 작성할 때 통합적으로 참조되는 조사 결과물을 산출한다.

> **구현 방향**: Claude Code의 Task Management System, fork, agent-teams, Agent Swarm, orchestrator agent, sub agents, skills, hooks, commands, task verification 등을 적절히 조합하여 `/prompt/workflow.md`의 워크플로우를 구현한다.

## Fork-Based Sessions Branch Strategy

이 프레임워크는 `/fork` 기능을 활용하여 **코딩 기법의 여러 경로를 동시에 탐색**합니다.

### 전략 개요

```
초기 설정
  ↓
[PHASE 1: 코딩 심층 조사 - 10개 Branch 병렬 실행]
  ├─ /fork Branch 1.1: Workflow Script Architect (선언적 워크플로우)
  ├─ /fork Branch 1.2: Workflow Script Architect (절차적 워크플로우)
  ├─ /fork Branch 2.1: Agent Orchestration Coder (중앙 집중 오케스트레이션)
  ├─ /fork Branch 2.2: Agent Orchestration Coder (분산 오케스트레이션)
  ├─ /fork Branch 3.1: Skills & Hooks Developer (범용 skill 라이브러리)
  ├─ /fork Branch 3.2: Skills & Hooks Developer (워크플로우 특화 skill)
  ├─ /fork Branch 4.1: Verification & Quality Coder (엄격한 검증)
  ├─ /fork Branch 4.2: Verification & Quality Coder (선택적 검증)
  ├─ /fork Branch 5.1: State & Recovery Coder (파일 기반 상태 관리)
  └─ /fork Branch 5.2: State & Recovery Coder (구조화된 상태 머신)
  ↓
[PHASE 2: 코딩 토론]
  ├─ /fork Branch 2.A: 워크플로우 표현력 우선 토론
  ├─ /fork Branch 2.B: 구현 안정성 우선 토론
  ├─ /fork Branch 2.C: 구현 속도 우선 토론
  └─ /fork Branch 2.D: 유지보수/확장성 우선 토론
  ↓
[PHASE 3: 구현 시나리오]
  ├─ /fork Branch 3.A: Full-Defensive (모든 엣지 케이스 처리)
  ├─ /fork Branch 3.B: Balanced (핵심만 방어)
  └─ /fork Branch 3.C: Rapid-Prototype (최소 구현, 빠른 검증)
  ↓
[PHASE 4: 조사 결과 통합]
  → 코딩 심층조사 종합 보고서
  → 구현 패턴 레퍼런스
  → 추가 조사가 필요한 발견 사항
```

### Branch 사용의 목적

**조사 Phase에서의 Branch**
- 같은 구성 요소도 구현 접근에 따라 복잡도와 안정성이 달라짐을 검증
- 예: "orchestrator가 모든 흐름 제어" vs "agent들이 자율 협업" → 같은 기능인데 코드 구조가 전혀 다름
- 각 접근의 장단점을 의사 코드 또는 구조 예시로 구체적으로 비교

**토론 Phase에서의 Branch**
- 어떤 코딩 철학을 우선시하냐에 따라 구현 패턴이 달라짐을 검증
- 각 경로의 코드 복잡도-안정성-유지보수성 트레이드오프를 파악

**시나리오 Phase에서의 Branch**
- 방어 수준에 따른 3가지 구현 경로 동시 검증
- 각 시나리오의 총 구현량, 안정성, 유지보수 부담을 비교

---

## 실행 제약 (Constraints)

**조사 단계 제약**
- 각 구현 패턴에 **의사 코드 또는 구체적 구조 예시**를 반드시 포함
- Claude Code의 실제 기능을 근거로 할 것 — 존재하지 않는 API/기능을 가정하지 않음
- **파킹 로트 규칙**: 조사 중 발견했으나 현재 Branch 범위를 넘는 사항은 파킹 로트에 기록

**토론 단계 제약**
- 토론은 **구현 패턴의 비교**에 집중 — "이 패턴이 이 구성 요소에 적합한가?"
- 모든 비교에 **코드 수준의 구체적 차이**를 명시 (추상적 "더 좋다" 금지)

**시나리오 단계 제약**
- 각 시나리오는 워크플로우 구현에 필요한 **모든 영역(스크립트, 오케스트레이션, skill/hook, 검증, 상태 관리)의 구현 구성**을 포함해야 함
- 각 시나리오에서 **구현 복잡도 분석**을 반드시 출력

---

## 성공 지표 (Success Points)

### Phase 1 (코딩 조사) 성공 지표

✅ **구현 기법이 구체적인가?**
- [ ] 각 구현 패턴에 대해 **의사 코드 또는 구조 예시**가 있는가?
- [ ] workflow.md의 실제 구조가 설계되었는가?
- [ ] orchestrator-sub agent 역할 분배가 구체적인가?

✅ **구현 복잡도 분석이 현실적인가?**
- [ ] 각 구현 패턴의 복잡도가 구체적으로 분석되었는가?
- [ ] 예상보다 쉬운 것과 어려운 것이 구분되었는가?

### Phase 2 (토론) 성공 지표

✅ **구현 패턴 간 트레이드오프가 명확한가?**
- [ ] 최소 3개 이상의 구현 방식 충돌 포착됨
- [ ] 각 충돌에 대해 코드 수준의 구체적 차이가 명시됨

### Phase 3 (시나리오) 성공 지표

✅ **3가지 구현 시나리오가 현실적으로 구분되는가?**
- [ ] 각 시나리오의 총 구현량, 안정성, 유지보수 부담이 다른가?
- [ ] 각 시나리오의 구체적 구현 구성이 명시되었는가?

### Phase 4 (통합) 성공 지표

✅ **종합 보고서가 최종 PRD 작성 시 유의미한 참조 자료인가?**
- [ ] 구현 패턴 레퍼런스가 실제 개발에 바로 사용 가능한 수준인가?
- [ ] 구현 과정에서 발견된 기술적 한계·가능성이 명확히 정리되었는가?
- [ ] 추가 조사가 필요한 발견 사항이 정리되었는가?

## 최종 검수 체크리스트

```
조사 단계:
[ ] 각 Branch가 구체적 구현 기법을 의사 코드/구조 예시와 함께 제시하는가?
[ ] 의사 코드 또는 구조 예시가 포함되어 있는가?
[ ] workflow.md의 실제 구조가 설계되었는가?

토론 단계:
[ ] 최소 3개 이상의 구현 패턴 트레이드오프가 명시되는가?
[ ] 코드 수준의 구체적 차이가 비교되는가?

시나리오 단계:
[ ] 3개 시나리오가 각각 다른 구현 구성을 제시하는가?
[ ] 각 시나리오의 총 구현량, 안정성, 유지보수 부담이 비교되는가?

최종:
[ ] 종합 보고서가 다른 심층조사 결과와 함께 읽을 때 유의미한가?
[ ] 구현 패턴 레퍼런스가 실제 개발에 사용 가능한 수준인가?
[ ] 조사 중 발견된 새로운 기술적 한계·가능성이 정리되었는가?
```

## PHASE 1: 코딩 심층 조사 (Fork-Based 병렬 탐색)

### 각 팀별 2개 Branch 병렬 실행

각 Teammate는 자신의 코딩 영역 내에서 **상충하는 2가지 구현 접근**을 동시에 탐색합니다.
탐색의 대상은 추상적 기법이 아니라 **구체적 워크플로우를 어떻게 구현하는가**입니다.

#### **Workflow Script Architect - 2개 Branch**

**Branch 1.1: Declarative Workflow (선언적 워크플로우)**

```
당신은 workflow.md를 **"무엇을 할지"만 명시하고 "어떻게"는 orchestrator에게 맡기는** 방식으로 설계하는 전문가입니다.

관점: "워크플로우는 의도만 선언하고, 실행 방법은 에이전트가 판단한다."

분석 대상: [리서치 주제]

분석 내용:

1. workflow.md 구조 설계
   - 태스크 정의 형식: [각 구성 요소를 태스크로 어떻게 표현하는가?]
   - 태스크 간 의존성 표현: [선후 관계, 병렬 가능 여부를 어떻게 명시하는가?]
   - 완료 조건(exit criteria) 표현: [각 태스크의 성공 기준을 어떻게 명시하는가?]
   - 구조 예시: [실제 workflow.md 샘플 3개 태스크 분량 작성]

2. orchestrator agent 역할 설계
   - orchestrator가 workflow.md를 읽고 판단해야 할 것: [실행 순서, 에이전트 배정, 리소스 할당]
   - orchestrator에게 필요한 지시사항: [CLAUDE.md에 어떤 내용이 필요한가?]
   - orchestrator의 자율 판단 범위: [어디까지 맡기고 어디서 제약하는가?]

3. 구현 복잡도 분석
   - 선언적 workflow.md의 작성 난이도: [낮음/중간/높음]
   - orchestrator 설정의 복잡도: [낮음/중간/높음]
   - 예상치 못한 동작의 가능성: [낮음/중간/높음]
   - 디버깅 난이도: [낮음/중간/높음]

결론:
- 선언적 접근의 강점: [구체적]
- 선언적 접근의 한계: [어떤 구성 요소에서 불충분한가?]
- 이 접근에서 필수적인 구현 요소: [Top 3]

🅿️ 파킹 로트: [기술 심층조사 결론과의 불일치, 범위 밖 발견 사항]
```

```

   - 실행 예측 가능성: [높음]

결론:
- 절차적 접근의 강점: [구체적]

🅿️ 파킹 로트: [불일치, 범위 밖 발견]
```

#### **Agent Orchestration Coder - 2개 Branch**

**Branch 2.1: Centralized Orchestration (중앙 집중)**

```
당신은 **orchestrator agent가 모든 흐름을 제어하고 sub agents에게 개별 태스크를 위임**하는 패턴의 전문가입니다.

   - 결과 병합 방법: [orchestrator가 어떻게 종합하는가?]

4. 구현 복잡도 분석
   - orchestrator 설정 자체의 복잡도: [__]
   - sub agent 수 증가 시 관리 부담: [__]
   - 단일 장애점(orchestrator 실패) 대응: [__]

결론:
- 중앙 집중의 강점: [예측 가능, 디버깅 용이]
- 중앙 집중의 한계: [orchestrator 과부하, 단일 장애점]
```

**Branch 2.2: Distributed Orchestration (분산 오케스트레이션)**

```
당신은 **Agent Swarm 패턴으로 에이전트들이 자율적으로 협업**하는 구현의 전문가입니다.
```

## 팁 & Best Practices

**PHASE 1 (조사 Branch) 운영 팁**
- **의사 코드 또는 구조 예시를 반드시 포함** — "가능하다"는 추상적 주장 금지
- Claude Code의 Task Management System, fork, agent-teams, Agent Swarm 등의 **실제 동작을 근거로** 설계

**PHASE 2 (토론 Branch) 운영 팁**
- 추상적 패턴 비교가 아니라 **구체적 워크플로우에 적용했을 때의 차이**를 논의
- 코드 수준의 구체적 차이를 반드시 명시

**PHASE 3 (시나리오 Branch) 운영 팁**
- 각 시나리오의 총 구현량, 구현 난이도, 토큰 소비, 유지보수 부담을 정량적으로 비교할 것

**PHASE 4 (최종 통합) 운영 팁**
- **구현 패턴 레퍼런스는 실제 개발에 바로 사용 가능한 수준**으로 작성
- 발견된 기술적 한계와 가능성을 솔직하게 정리 — 최종 PRD 작성 시 핵심 참조 자료
- 추가 조사가 필요한 사항을 명확히 정리
