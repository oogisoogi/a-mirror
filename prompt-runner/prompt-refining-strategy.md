# 프롬프트 정밀 재설계 전략
> 목적: 이 문서를 클로드 코드에게 제공하고, 기존 프롬프트 블럭 전체를 이 기준으로 다시 정밀하게 깎는다.

---

## 1. 이 전략이 만들어진 배경

### 진단된 구조적 문제

실제 112개 프롬프트 전수 분석 결과:

- 전체 프롬프트의 **73%가 성찰/검증**. 실행 지시는 15%.
- "workflow 철학 보존", "RLM 패턴 유지", "품질 절대기준" 등 핵심 기준이 **20개 가까운 프롬프트에서 반복** 선언됨.
- 절대 기준이 매 프롬프트 **중간이나 하단**에 주의사항으로 붙어 있음.
- 성찰이 깊이를 쌓는 게 아니라 **동일 층위를 반복** 순환함.

### 핵심 원인

클로드가 매 턴 방향을 잃기 때문에, 매 턴 다시 잡아줘야 하는 구조다.
결과적으로 성찰 위에 성찰을 쌓는 악순환이 만들어진다.

### 왜 CLAUDE.md만으로 해결이 안 되는가

CLAUDE.md는 클로드에게 **배경 맥락**으로 처리된다. "지금 이 턴의 절대 기준"으로 작동하지 않는다.
반면 프롬프트 안에 직접 넣은 지시는 **지금 이 턴의 활성 컨텍스트**에 있다. 이것이 절대적 각인처럼 작동한다.

### 결론: 두 가지가 모두 필요하다

- **CLAUDE.md**: 프로젝트 전체 철학과 구조의 배경 기준
- **프롬프트 내 ABSOLUTE ANCHOR**: 해당 턴의 절대 기준을 각인시키는 역할

## 2. ABSOLUTE ANCHOR — 핵심 설계 원칙

### 왜 현재 방식이 비효율인가

현재 방식은 절대 기준을 **서술형으로 길게 반복**한다.

```
// 현재 방식 예시 (~450자)
단, 모든 과정에서 절대적 기준도 잊지 말아야 한다.
해당 워크플로우가 작동할 때 속도와 토큰 비용은 완전히 무시하라.
내가 원하는 절대 기준은 최종결과물의 품질과 질적 수준이다.
다시 말한다. 품질과 최고 수준의 질적 결과물이 절대 기준이다.
... (이하 동일 내용 반복)
```

문제는 분량이 아니다. **핵심이 반복 속에 묻히는 것**이 문제다.
그리고 왜 이 기준이 절대적인지, 충돌 시 어떻게 판단하는지가 빠져 있다.

### ABSOLUTE ANCHOR의 3가지 필수 요소

압축된 키워드만으로는 클로드가 **트레이드오프 상황에서 판단 근거**를 갖지 못한다.
아래 세 요소가 모두 있어야 각인이 작동한다.

1. **키워드** — 무엇이 기준인가
2. **이유** — 왜 이 기준이 절대적인가
3. **충돌 시 판단** — 트레이드오프 상황에서 어떻게 결정하는가

### 표준 ABSOLUTE ANCHOR 템플릿

```
[ABSOLUTE ANCHOR]
① 품질 절대우선 — 속도/비용/편의는 이유가 될 수 없다.
   트레이드오프 상황에서 품질 외 모든 것은 희생 가능하다.

② SOT + RLM 불변 — 어떤 개선도 이 두 구조를 침범할 수 없다.
   침범 가능성이 보이면 구현 전 반드시 나에게 먼저 알려라.

③ 승인 후 구현 — 단 한 줄의 코드도 내 승인 없이 작성 금지.
   승인 요청 시: 구현 의도(1문장) + 영향 파일 + 위험 지점만 보고하라.
[/ABSOLUTE ANCHOR]
```

**목표 분량**: 150자 내외. 현재 방식 대비 65% 감소.

### ABSOLUTE ANCHOR의 위치

반드시 **프롬프트 맨 앞**에 배치한다.

클로드는 프롬프트 앞부분에 주의가 가장 집중된다.
앞에 오면 클로드가 작업을 읽기 전에 이미 판단 기준을 탑재한 상태로 시작한다.

```
[ABSOLUTE ANCHOR]
... (기준)
[/ABSOLUTE ANCHOR]

[작업 지시]
... (실제 작업 내용)
```

## 3. 프롬프트 재설계 7가지 기준

### 기준 1. ABSOLUTE ANCHOR는 프롬프트 상단에, 항상

모든 프롬프트는 ABSOLUTE ANCHOR로 시작한다.
단, 프로젝트 성격에 따라 ①②③의 내용은 조정 가능하다.
형식과 위치는 고정이다.

### 기준 2. 절대 기준의 반복 선언을 제거하고 위계로 정리

현재: 동일한 기준을 매 프롬프트에서 서술형으로 반복
개선: ABSOLUTE ANCHOR 한 블럭으로 통일, 내부에 위계 구조 적용

**위계 구조 원칙**
충돌이 발생할 때 상위 기준이 절대 우선한다.
클로드가 위계를 알면, 모호한 상황에서 스스로 판단한다.

```
[ABSOLUTE ANCHOR]
제1 기준 (절대 불변): 결과물의 품질
제2 기준 (구조 불변): SOT + RLM 패턴
제3 기준 (프로세스): 구현 전 승인
→ 상위 기준과 충돌하는 하위 기준의 실행은 중단하고 보고하라.
[/ABSOLUTE ANCHOR]
```

### 기준 3. 성찰은 층위를 올리는 구조로

현재: 동일 층위 성찰 반복 (세션당 성찰 12~13개)
개선: 3층위로 분리, 각 층위가 다음 층위의 입력이 됨

```
[1층위 — 사실 확인 (Fact Check)]
구현된 것이 설계 의도와 일치하는가?
맞다/틀리다만 판단. 이유 설명 금지.
출력: 불일치 항목 목록만.

[2층위 — 구조 분석 (Structure Analysis)]
1층위 불일치를 가지고:
6개월 후 기능 추가 시나리오 2개 설정 →
현재 구조에서 어디가 먼저 무너지는지 분석.
출력: 취약 지점과 이유.

[3층위 — 철학 정합성 (Philosophy Alignment)]
2층위 취약 지점을 가지고:
이것이 제1 기준(품질)을 실제로 훼손하는가?
"좋은 것 같다"가 아닌 구체적 증거로 판단.
출력: 수정 필요 / 수정 불필요 / 판단 불가 + 이유.
```

**효과**: 기존 5개 성찰 프롬프트를 3개로 통합하면서, 각 성찰이 이전 성찰의 결과를 실제로 올라타는 구조가 됨.

### 기준 4. 적대적 에이전트는 공격 축을 전문화

현재: "비판적 사고로 공격"이라는 단일 지시
개선: 에이전트별 전문화된 공격 축 부여

```
[적대적 에이전트 A — 운영 실패 전문가]
공격 무기: 동시성 충돌 / 에이전트 응답 불능 / SOT 데이터 불일치 /
           네트워크 단절 / 메모리 초과 / 재귀 무한루프
찾는 결함: "작동은 하지만 실패 시 복구 불가능한 부분"

[적대적 에이전트 B — 단순성 원리주의자]
공격 무기: 더 단순한 대안 / 오버엔지니어링 지점 /
           제거해도 동일 품질이 나오는 컴포넌트
찾는 결함: "복잡성을 위한 복잡성"

[적대적 에이전트 C — 6개월 후 유지보수자]
공격 무기: 이해 불가능한 구조 / 문서 없는 가정 /
           변경 시 어디를 건드려야 할지 모르는 부분
찾는 결함: "지금은 작동하지만 미래에 함정인 부분"

[방어 규칙]
방어 불가능한 지점은 솔직하게 인정하라.
방어 불가능 = 수정해야 할 핵심.
인정 없이 강변하는 것은 금지.
```

### 기준 5. 읽기 프로토콜에 역방향 층을 추가

현재: 3단계 읽기
개선: 4단계 읽기 — 4차에 "없는 것을 찾는" 역방향 읽기 추가

```
1차: 본질(What/Why) 집중. 목적/철학/방향만 파악.
2차: 연관 확장. 시스템 요구사항/단계/기능 파악.
3차: 중간 부분 정밀 보충.
4차: [역방향] 이 문서가 다루지 않는 것이 무엇인가?
     사용자의 목적 달성에 필요하지만 이 문서에 없는 것을 찾아 목록화하라.
     있는 것을 확인하는 읽기가 아니다. 없는 것을 찾는 읽기다.
```

### 기준 6. 승인 게이트를 보고 형식으로 구조화

현재: "주의: 최종 구현은 나의 승인을 받고 시작해야 한다" — 19개 프롬프트 반복
개선: 보고 형식을 ABSOLUTE ANCHOR 안에 명시. 개별 프롬프트에서 "주의" 문장 제거.

```
승인 요청 필수 형식 (이 형식 없이 단 한 줄의 코드도 작성 금지):
- [구현 의도] 한 문장: 이 작업이 무엇을 하는가
- [영향 파일] 수정될 파일 목록과 각각의 변경 이유
- [위험 지점] 실패 시 파급 효과가 큰 부분
- [가정 사항] 모호해서 내가 판단한 부분
- [보존 확인] SOT/RLM 유지 여부: 예/아니오 + 근거
```

### 기준 7. 세션 시작 프롬프트는 헤더 형식으로 표준화

현재: 세션마다 배경 설명을 새로 서술 → 첫 프롬프트 1,000~2,500자
개선: SESSION HEADER 형식으로 통일 → 100자 이내

```
[SESSION HEADER]
이전 단계: (완료된 산출물 명시)
이번 단계: (이번 세션의 목표 한 줄)
핵심 참조 파일: (경로 목록)
[/SESSION HEADER]
```

ABSOLUTE ANCHOR가 기준 전체를 담고 있으므로,
세션 헤더는 "지금 어느 단계인가"만 담으면 된다.

## 5. 각인 효과 검증 기준

재설계한 프롬프트가 실제로 각인됐는지 확인하는 기준이다.

**검증 포인트 1 — SOT/RLM 자발적 언급**
클로드가 SOT/RLM 충돌 가능성을 스스로 감지하고 먼저 알려오는가.
이것이 나오면 각인 성공. 나오지 않으면 ② 설명 강화 필요.

**검증 포인트 2 — 품질 타협 제안 없음**
"이 방법이 더 빠르긴 한데..." 같은 문장이 나오면 ① 각인 실패.

**검증 포인트 3 — 승인 게이트 자발적 작동**
지시 없이도 "구현 전 승인 요청드립니다 + 형식 보고"가 나오면 ③ 각인 성공.

## 6. 클로드 코드에게 주는 작업 지시

이 문서를 읽은 클로드 코드는 아래 기준으로 기존 프롬프트 블럭을 재설계한다.

### 작업 원칙

1. **각 프롬프트의 목적을 먼저 파악하라.**
   성찰인지 / 실행 지시인지 / 아이디어 회의인지 / 통합 정리인지.
   목적이 다르면 ABSOLUTE ANCHOR의 ①②③ 강조점도 달라진다.

2. **반복되는 절대 기준 서술을 ABSOLUTE ANCHOR로 교체하라.**
   키워드 + 이유 + 충돌 시 판단, 이 세 요소가 모두 있어야 한다.
   그 외 반복 선언은 전부 제거한다.

3. **성찰 프롬프트는 층위를 명시하라.**
   1층위(사실 확인) → 2층위(구조 분석) → 3층위(철학 정합성).
   같은 층위가 연속으로 나오면 통합 가능한지 검토하라.

4. **적대적 에이전트 프롬프트는 공격 축을 전문화하라.**
   "비판적으로 공격"을 에이전트 A/B/C 역할 분리로 교체하라.

5. **읽기 프로토콜이 있는 프롬프트는 4차 역방향 읽기를 추가하라.**

6. **세션 시작 프롬프트는 SESSION HEADER 형식으로 교체하라.**

7. **"주의: 최종 구현은 나의 승인을 받고" 문장은 전부 제거하라.**
   ABSOLUTE ANCHOR ③에서 이미 처리된다.

8. **재설계 후 각 프롬프트의 목적이 한 문장으로 설명 가능한지 확인하라.**
   설명이 안 되면 프롬프트가 아직 정리되지 않은 것이다.

### 재설계 금지 사항

- 프롬프트의 핵심 목적을 바꾸는 것
- 사용자의 실제 작업 의도를 축약하면서 내용을 잃는 것
- ABSOLUTE ANCHOR를 작업 지시 뒤에 배치하는 것
- "좋은 것 같다" 수준의 모호한 표현을 남기는 것
